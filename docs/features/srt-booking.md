# SRT 예매 자동화 (srt-booking)

## 개요
SRT(수서고속철도) 공식 예매 시스템에서 원하는 일정의 잔여석을 주기적으로 확인하고, 좌석이 풀리는 순간 자동으로 결제 단계 직전까지 진행하는 자동화 기능.

## 목표
- 매진된 인기 시간대 열차의 취소표를 사람보다 빠르게 잡는다.
- 잡은 직후 사용자에게 알림을 보내고, 결제만 사용자가 직접 처리한다.
- 운영자가 여러 일정을 동시에 감시할 수 있다.

## 비목표 (Out of Scope)
- 결제 자동화 (PG 약관 위반 소지). 결제 직전 단계까지만 자동화.
- KTX(코레일) 예매. 별도 모듈로 분리.
- 단체 예매 / 정기권 / 자유석.

## 사용자 스토리
1. 사용자는 출발역, 도착역, 날짜, 시간 범위, 좌석 등급(일반/특실), 인원을 입력해 감시 작업을 등록한다.
2. 시스템은 SRT 사이트의 조회 페이지를 N초 간격으로 폴링한다.
3. 매칭되는 좌석이 발견되면 즉시 좌석 선택 → 임시 예약 단계까지 진행한다.
4. 사용자에게 알림(텔레그램/슬랙/이메일 중 1개)을 보내고, 결제 페이지 URL과 임시 예약 만료 시각을 전달한다.
5. 사용자가 N분 내에 결제를 완료하지 않으면 임시 예약이 자동 만료된다.

## 핵심 입력
| 항목 | 타입 | 예시 |
|---|---|---|
| 출발역 | string | 수서, 동탄, 평택지제, 천안아산, 오송, 대전, 동대구, 부산 |
| 도착역 | string | 위와 동일 |
| 날짜 | YYYY-MM-DD | 2026-06-15 |
| 출발 시간 범위 | HH:MM ~ HH:MM | 08:00 ~ 11:00 |
| 좌석 등급 | enum | general / first / any |
| 인원 | int | 1~9 |
| 폴링 간격 | int (sec) | 기본 10초, 최소 5초 |
| 알림 채널 | enum | telegram / slack / email |

## 아키텍처
- **scraper**: SRT 예매 페이지 세션 관리, 로그인, 조회, 좌석 선점. Playwright(또는 requests + 세션 쿠키) 기반.
- **scheduler**: 등록된 작업을 주기적으로 실행하는 워커. APScheduler 또는 단순 asyncio 루프.
- **notifier**: 좌석 확보 시 알림 전송. 채널별 어댑터 패턴.
- **store**: 작업 목록과 실행 상태 저장. 초기 버전은 SQLite, 확장 시 Postgres.
- **cli / api**: 작업 등록·취소·상태 조회. 초기 버전은 CLI, 이후 FastAPI 추가.

## 비밀 관리
- SRT 회원번호·비밀번호는 OS 키링 또는 `.env` (`SRT_ID`, `SRT_PW`)에 저장. 저장소에 절대 커밋 금지.
- 알림 채널 토큰도 동일하게 `.env`.

## 컴플라이언스 / 운영 주의
- SRT 약관상 자동화 도구 사용에 대한 명시적 금지 조항이 있는지 사전 확인 필요. **확실하지 않음** — 운영 전 약관 재확인.
- 과도한 폴링(초당 다수 요청)은 IP 차단 위험. 기본 10초, 최소 5초 가드.
- 동일 계정으로 동시 작업 다수 실행 금지 (세션 충돌).
- 캡차가 노출되면 자동 진행을 멈추고 사용자 개입 요청.

## 마일스톤
1. **M1 — 스크래퍼 PoC**: 수동 입력 한 건에 대해 로그인 → 조회 → 좌석 선점까지 통과.
2. **M2 — 스케줄러**: 단일 작업을 백그라운드 폴링.
3. **M3 — 알림**: 텔레그램 1채널 연동.
4. **M4 — 다중 작업 + CLI**: 작업 등록/취소/상태 조회.
5. **M5 — 운영 안정화**: 캡차 감지, 세션 만료 복구, 로깅, 알림 채널 확장.

## 검증 기준
- M1: 좌석 있는 시간대에 대해 임시 예약 성공 (수동 1회 확인).
- M2: 매진 시간대 작업 등록 후 좌석 풀리는 순간 30초 이내 선점.
- M3: 선점 직후 5초 이내 텔레그램 알림 도착.
- M4: 동시 5개 작업을 안정적으로 운영 (24시간).

## 미해결 질문
- SRT 로그인 단계의 캡차/2FA 정책 현황 (재확인 필요).
- 임시 예약 유지 시간 (10분으로 알고 있으나 공식 확인 필요).
- 결제 단계까지 URL 공유로 사용자 인계가 가능한지 (세션 쿠키 분리 이슈).
